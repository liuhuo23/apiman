# 子模块 02 - 路由骨架与页面布局 - 计划

## 1. 目标

在前端工程壳层已完成的基础上，建立应用的路由骨架和页面布局框架。

本子模块的目标是先把“页面怎么切、区域怎么分、路由怎么挂”定下来，不进入具体业务内容。

## 2. 范围

### 2.1 包含内容

1. 应用级路由结构设计。
2. 主页、项目详情页、项目设置页的路由入口。
3. 项目详情页的左右分栏布局。
4. 项目详情页内部子路由出口。
5. 路由占位页面和布局容器。
6. 与桌面应用相适配的导航层级。

### 2.2 不包含内容

1. 项目列表数据加载。
2. API 树数据加载。
3. 接口文档编辑。
4. 接口测试执行。
5. 项目设置的真实表单内容。
6. 本地持久化实现。
7. Rust 命令实现。

## 3. 输入

1. 已完成的前端工程壳层。
2. 已确定的页面目标与架构说明。
3. 当前使用的 React + TypeScript + Ant Design 技术栈。

## 4. 输出

1. 可挂载后续页面的路由骨架。
2. 明确的页面布局层次。
3. 可复用的布局容器。
4. 路由预留出口和导航切换方式。

## 5. 依赖关系

本子模块依赖：

1. 子模块 01 已完成并可正常启动。
2. 页面壳层已提供统一主题和基础布局。

本子模块会为后续模块提供：

1. 页面入口位置。
2. 子路由挂载位置。
3. 详情页布局容器。
4. 全局导航框架。

## 6. 风险点

1. 路由层级一开始设计得过深，后续页面会难以扩展。
2. 主页和项目详情页之间的职责边界不清，容易导致布局重复。
3. 项目详情页如果把文档、测试、设置混在同一层，后续维护成本会变高。
4. 子路由设计过早耦合业务数据，会阻塞后续模块拆分。

## 7. 不做事项

1. 不在本阶段实现数据请求。
2. 不在本阶段实现接口树或项目列表。
3. 不在本阶段实现任何编辑、测试和保存逻辑。

## 8. 分阶段计划

1. 确定全局路由树。
2. 定义项目详情页的布局结构。
3. 定义项目详情页子路由出口。
4. 预留主页、设置页和详情页的页面容器。
5. 补充最小可运行的路由跳转。
6. 完成文档审查后再进入实现。

## 9. 验收标准

该子模块的计划阶段完成后，应满足：

1. 范围清楚。
2. 路由职责清楚。
3. 布局职责清楚。
4. 与后续子模块的边界清楚。
5. 允许进入设计说明审查。
