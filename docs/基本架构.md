# 基本架构说明

## 1. 项目定位

本项目是一个桌面端 API 管理工具，核心目标是帮助用户管理 API 项目、浏览接口结构、维护接口文档，并在本地完成接口测试。

设计原则：

1. 前端负责交互、展示和基础状态管理。
2. 后端 Rust 负责核心业务、安全能力和数据处理。
3. 数据不依赖数据库，优先通过本地文件系统持久化。
4. UI 采用统一主题，支持明暗两套外观。

## 2. 路由设计

项目使用 React Router 组织页面，整体按桌面应用的页面切换方式设计。

### 2.1 主页

主页是应用入口，进入应用后默认展示项目列表，并提供创建项目入口。

页面职责如下：

1. 展示所有 API 项目列表。
2. 支持创建新项目。
3. 支持卡片多选。
4. 支持点击项目卡片进入项目详情页。
5. 作为全局入口，承担项目的快速切换能力。

主页中的项目以 Card 形式展示，单个 Card 需要包含：

1. 项目名称。
2. 创建时间。
3. 最新更新时间。
4. 右下角删除图标。
5. 卡片选中状态。

项目卡片上的图标应优先使用 Ant Design 推荐图标和 React 常用交互写法。

创建项目时需要弹出模态框，模态框内支持多种导入方式：

1. URL 方式，支持标准 Swagger / OpenAPI 地址导入。
2. 文件上传方式，目前仅支持 `openapi.json` 和 `yaml`。
3. 跳过导入，先创建空项目。

### 2.2 项目详情页

项目详情页采用左右分栏布局：

1. 左侧为 API 树形列表，用于浏览接口分组和接口节点。
2. 右侧为接口详情区域，用于展示当前接口的信息、配置和操作。
3. 接口详情区域下继续拆分子路由，保持页面职责清晰。
4. 页面顶部应提供返回主页入口，方便从深层页面退出到项目列表。

接口详情的子路由建议包括：

1. `文档`：展示接口文档，支持手动编辑，也支持从 `swagger` / `openapi.json` 解析生成。
2. `测试`：用于发送实际 API 请求并查看响应结果。

### 2.3 应用设置页

应用设置页用于维护全局配置，主要包括：

1. 通用参数配置。
2. 多环境配置，例如测试环境、生产环境等。
3. 与项目运行、请求测试相关的基础选项。
4. 主题模式切换。
5. 命令模式切换。
6. 页面顶部应提供返回主页入口。

### 2.4 删除项目

主页中的项目卡片支持删除操作。

删除操作应当遵循以下原则：

1. 删除入口固定在卡片右下角。
2. 删除动作与卡片点击进入详情页的行为分离。
3. 执行删除前需要二次确认。
4. 支持对多选后的项目批量删除。

## 3. 设置页

设置页是应用级配置入口，负责管理不属于项目本身的全局行为。

设置页主要承担以下职责：

1. 主题模式切换。
2. 命令模式切换。
3. 本地应用偏好配置。
4. 其他未来需要统一管理的桌面偏好项。

设置页的状态变化应立即反映到当前应用，并同步保存到本地配置。

## 4. 代码结构

代码结构需要尽量清晰，避免跨层耦合。

推荐目录约定如下：

1. `src/pages/`：存放所有路由页面。
2. `src/components/`：存放通用自定义组件。
3. `src/types/`：存放数据模型和类型定义。
4. `src/services/`：前端对后端命令、接口和文件能力的调用封装。
5. `src/stores/`：全局状态管理。
6. `src/utils/`：通用工具方法。
7. `src/styles/`：全局样式、主题变量和视觉规范。
8. `src-tauri/src/`：Rust 后端业务、命令和本地能力实现。

建议的页面结构如下：

1. `pages/HomePage`：主页。
2. `pages/ProjectLayout`：项目详情页外层布局。
3. `pages/ProjectDocsPage`：接口文档页。
4. `pages/ProjectTestPage`：接口测试页。
5. `pages/AppSettingsPage`：应用设置页。

建议的后端结构如下：

1. `src-tauri/src/main.rs`：程序入口。
2. `src-tauri/src/lib.rs`：应用启动和 Tauri 容器初始化。
3. `src-tauri/src/commands/`：暴露给前端调用的 Rust 命令。
4. `src-tauri/src/services/`：文件读写、接口解析、请求执行等领域服务。
5. `src-tauri/src/models/`：后端数据结构定义。
6. `src-tauri/src/storage/`：本地文件系统持久化实现。

## 4. 模块职责

### 4.1 前端职责

前端应承担以下职责：

1. 页面展示与交互。
2. 路由组织与页面切换。
3. 表单输入、局部状态和视图状态管理。
4. 调用后端命令并渲染返回结果。

前端不应承担以下职责：

1. 直接操作敏感文件结构。
2. 直接执行复杂的业务规则。
3. 自行拼装无法复用的数据处理逻辑。

### 4.2 后端职责

后端 Rust 应承担以下职责：

1. 项目文件的读取、写入和校验。
2. 接口文档解析与数据规范化。
3. API 请求执行与响应包装。
4. 环境配置和全局参数管理。
5. 需要保护的安全逻辑和敏感能力。

### 4.3 共享类型职责

`types/` 中的类型应作为前后端协作协议的基础：

1. 定义项目、接口、环境、文档等领域模型。
2. 定义前后端通信的数据结构。
3. 尽量保持字段稳定，避免不同模块各自定义同一套结构。

## 5. 数据流

系统内的数据流应尽量保持单向、可追踪。

### 5.1 项目加载流程

1. 前端进入主页后请求后端读取本地项目列表。
2. 后端从文件系统加载项目索引并返回给前端。
3. 前端渲染项目列表，用户可选择项目卡片、发起创建、删除或进入项目详情页。

### 5.2 首页创建流程

1. 用户点击主页上的创建按钮。
2. 前端弹出创建项目模态框。
3. 用户选择 URL 导入、文件上传或跳过导入。
4. 前端将创建请求提交给后端。
5. 后端根据导入方式生成或初始化项目数据。
6. 前端刷新项目列表并展示最新结果。

### 5.3 接口详情流程

1. 用户在左侧 API 树中选择接口。
2. 前端切换到对应接口详情视图。
3. 前端向后端请求接口完整定义、文档和环境信息。
4. 后端读取本地文件并返回规范化数据。
5. 前端根据当前子路由渲染文档或测试页面。

### 5.4 接口测试流程

1. 用户在测试页填写请求参数、请求头和请求体。
2. 前端将测试请求提交给后端。
3. 后端负责组装请求、执行调用并处理返回结果。
4. 前端展示响应内容、状态码、耗时和错误信息。

### 5.5 持久化流程

1. 用户修改项目、接口或环境配置。
2. 前端提交变更给后端。
3. 后端完成校验后写入本地文件系统。
4. 需要时再由后端重新读取并同步给前端。

## 6. 持久化方案

项目不使用数据库，而是通过系统文件系统完成本地持久化。

建议保存内容包括：

1. 项目配置。
2. 接口定义。
3. 环境参数。
4. 文档内容。
5. 请求测试相关的本地配置。

文件格式可按场景选择 `json` 或其他更适合的文档格式，具体实现时再确定。

## 7. 页面风格

前端 UI 统一使用 `antd + React` 作为基础框架。

视觉规范要求：

1. 全局样式统一。
2. 支持白天和夜间模式。
3. 背景、文字、阴影、边框等基础视觉元素需要有统一规范。
4. 组件风格要保持一致，避免页面间视觉割裂。
5. 每个页面的根容器最小尺寸必须不小于一个屏幕的宽高。

## 8. 桌面应用要求

本项目是桌面应用，不应直接套用纯 Web 项目的默认范式。

实现时需要重点关注：

1. 视图布局适配桌面窗口。
2. 页面切换要干净直接，不依赖全局头部来承载导航。
3. 深层页面应在页面内容区顶部提供局部返回入口，而不是依赖全局壳。
4. 操作路径尽量短，支持高频切换和快速浏览。
5. 页面结构适合多面板和多层级导航。

## 9. 组件设计原则

整体实现必须结构化拆分，避免过度耦合。

基本要求：

1. 单个组件职责单一。
2. 页面尽量只做组合，不承载复杂业务逻辑。
3. 状态、数据、视图分离。
4. 公共能力抽到通用层，避免重复实现。
5. 避免建立只服务于 Web 风格的全局壳层。

## 10. AI 与构建校验

开发完成后必须进行基础校验：

1. Rust 代码完成后执行 `cargo check`。
2. 前端代码完成后执行 `pnpm build`。

目标是尽量在提交前发现编译错误、类型问题和构建问题，降低引入明显 bug 的概率。
