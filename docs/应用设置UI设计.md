# 应用设置 UI 设计说明

## 1. 设计目标

设置页用于管理应用级行为，不承载项目业务内容。

本页主要目标：

1. 提供主题模式切换。
2. 提供命令模式切换。
3. 保持页面干净、直接、易理解。
4. 所有切换立即生效，并保存到本地配置。

## 2. 页面定位

设置页是独立页面，不是全局壳中的一个附属区域。

进入设置页后，用户应直接看到设置内容，而不是先经过一层公共头部或导航壳。

## 3. 页面结构

建议将设置页拆分为以下区域：

1. 页面标题区。
2. 主题设置区。
3. 命令模式设置区。
4. 其他应用偏好区。
5. 保存状态提示区。

### 3.1 页面标题区

顶部只保留必要的信息：

1. 页面标题。
2. 简短说明文字。
3. 返回主页按钮，放在标题区左侧，作为离开设置页的唯一明确入口。

### 3.2 主题设置区

主题设置区用于切换明暗模式。

建议包含：

1. 当前模式说明。
2. 主题切换控件。
3. 应用效果预览提示。

### 3.3 命令模式设置区

命令模式用于定义桌面应用的工作方式。

建议包含：

1. 当前命令模式说明。
2. 模式切换控件。
3. 对当前模式的解释文本。

### 3.4 其他应用偏好区

后续如果有需要统一控制的桌面行为，可以继续放在这里，例如：

1. 窗口行为偏好。
2. 默认启动行为。
3. 本地缓存偏好。

## 4. 交互原则

1. 页面切换后立即反馈当前状态。
2. 切换项尽量使用明确的单选或开关控件。
3. 避免嵌套过深的配置层级。
4. 不在设置页中放置项目业务操作。

## 5. 状态设计

设置页至少需要以下状态：

1. 当前主题模式。
2. 当前命令模式。
3. 本地保存状态。
4. 初始化加载状态。

## 6. 组件拆分建议

设置页建议拆分为以下组件：

1. `AppSettingsPage`：页面容器。
2. `ThemeSettingCard`：主题模式卡片。
3. `CommandModeCard`：命令模式卡片。
4. `PreferenceCard`：其他偏好卡片。

## 7. 视觉规范

设置页应保持克制，偏向信息清晰而不是装饰性。

建议使用：

1. Card 分区展示。
2. 清晰的单选或切换控件。
3. 明确的保存反馈。
4. 与首页和项目详情页一致的圆角、阴影和边框规范。

## 8. 保存逻辑

1. 用户修改设置。
2. 前端立即更新应用状态。
3. 前端将设置写入本地配置。
4. 下次启动时恢复上次设置。

## 9. 验收标准

该设计文档通过后，应满足：

1. 设置页是独立页面。
2. 主题和命令模式都在设置页管理。
3. 页面足够干净，不依赖全局壳。
4. 保存和恢复路径明确。
5. 可以进入实现阶段。
