# 子模块 02 - 路由骨架与页面布局 - 设计说明

## 1. 设计目标

建立桌面应用风格的路由与页面布局骨架，确保后续页面可以按模块独立扩展。

设计原则：

1. 路由先稳定，业务后补充。
2. 布局先统一，内容后细化。
3. 页面职责单一，避免页面同时承担导航、数据和编辑逻辑。

## 2. 路由结构

建议路由分为三类：

1. 主页路由。
2. 项目详情路由。
3. 项目设置路由。

### 2.1 主页路由

主页作为应用默认入口，职责是承接项目列表和全局切换入口。

在本阶段中，主页只需要具备以下结构能力：

1. 作为根路由入口。
2. 预留项目列表展示区域。
3. 预留跳转到项目详情页的交互出口。

### 2.2 项目详情路由

项目详情路由采用主布局 + 子路由的方式组织。

建议结构：

1. 左侧为项目内导航或 API 树占位区。
2. 右侧为详情内容区。
3. 详情内容区内再通过子路由切分为文档页、测试页等。

在本阶段中，项目详情页只完成布局和出口，不承载具体数据。

### 2.3 项目设置路由

项目设置路由用于承载项目级配置。

本阶段仅保留页面容器和布局出口，后续再添加表单和配置项。

## 3. 页面布局设计

### 3.1 全局布局

全局布局继续沿用子模块 01 的壳层设计，并在壳层内挂载路由内容。

布局要求：

1. 顶部保留统一窗口头部。
2. 中间区域承载路由内容。
3. 左右或上下布局按页面需要组合，但必须统一视觉规范。

### 3.2 项目详情页布局

项目详情页建议采用固定左栏 + 自适应右栏的桌面布局。

左栏职责：

1. 展示项目内结构入口。
2. 承载树形导航或列表导航。
3. 预留节点切换行为。

右栏职责：

1. 展示当前接口或设置的详情内容。
2. 承载子路由出口。
3. 保持足够的内容宽度，适合阅读和编辑。

### 3.3 子路由出口

详情区内部建议预留以下出口：

1. 文档页出口。
2. 测试页出口。
3. 其他扩展页出口。

这些出口在本阶段只做结构，不绑定实际业务内容。

## 4. 目录建议

本子模块完成后，前端建议形成如下结构：

1. `src/pages/HomePage/`
2. `src/pages/ProjectLayout/`
3. `src/pages/ProjectSettingsPage/`
4. `src/pages/ProjectDocsPage/`
5. `src/pages/ProjectTestPage/`
6. `src/router/`

说明：

1. `pages/` 只放路由页面。
2. `router/` 用于集中定义路由关系和跳转规则。
3. 通用布局组件继续放在 `components/` 中复用。

## 5. 数据与状态边界

本阶段不引入业务数据，仅保留最小状态边界：

1. 当前激活页面。
2. 当前激活子路由。
3. 左侧导航选中状态。

以下内容不在本阶段处理：

1. 项目数据加载。
2. 接口树加载。
3. 请求参数和响应数据。
4. 配置持久化数据。

## 6. 交互设计

建议的交互方式如下：

1. 在主页点击项目后进入详情页。
2. 在详情页左侧切换节点，右侧切换对应子路由。
3. 在设置页中按分区组织配置项。

交互上要保持桌面应用特征：

1. 路径短。
2. 切换快。
3. 内容区清晰。
4. 导航固定且易定位。

## 7. 实现边界

本子模块只允许实现以下能力：

1. 路由定义。
2. 布局容器。
3. 页面壳位。
4. 跳转占位。

本子模块禁止实现以下能力：

1. 数据请求。
2. 文件读写。
3. 表单保存。
4. 接口测试。
5. 文档解析。

## 8. 依赖和输出

本子模块依赖子模块 01 的壳层能力。

完成后输出给后续模块：

1. 页面入口结构。
2. 路由容器。
3. 详情页布局容器。
4. 子路由插槽。

## 9. 验收标准

该设计说明通过后，应该满足：

1. 路由层级明确。
2. 页面布局明确。
3. 页面职责明确。
4. 与后续业务模块的边界明确。
5. 可以进入实现阶段。
